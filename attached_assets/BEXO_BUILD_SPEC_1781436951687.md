# BEXO — Complete Build Specification

> A mobile-first portfolio builder for students & early-career professionals.
> Students sign in with WhatsApp OTP, complete a guided onboarding (resume upload + AI parsing, or manual entry), pick a theme/font/vibe, and get a live portfolio website at `{handle}.mybexo.com`.

This document is the **single source of truth** for rebuilding BEXO from scratch. It covers product purpose, tech stack, design system, database schema, every screen with its fields/validation/navigation logic, state management, services, and the profile-completion build gate.

---

## 1. Product Summary

**Name:** Bexo
**Platform:** Cross-platform mobile app (Expo / React Native), web-capable
**Core Loop:**

1. User signs in via phone number + WhatsApp OTP (or Google OAuth)
2. Completes onboarding: email → photo → handle → resume upload (AI) **or** manual entry → bio → DOB → theme → font → vibe preferences
3. Once profile reaches **90% completeness**, a portfolio site is auto-generated at `{handle}.mybexo.com`
4. User manages their live portfolio from a 3-tab dashboard: **Dashboard**, **Portfolio**, **Post Update**

**Target users:** Students and new grads building their first professional presence; early-career professionals updating credentials on mobile. Context = "phone in daylight" — calm, trustworthy, NOT a late-night dev tool.

**Tone:** Warm, clear, encouraging — like a patient editor helping you publish. No "elevate," "seamless," "unlock your potential" filler language.

**Anti-references (do NOT build like this):**
- Purple AI-gradient SaaS dashboards
- Dark-only operator UIs (this is light/editorial)
- Generic 3-column stat-card grids
- Fake progress bars that finish before the real build does
- Modal-heavy flows where inline progressive disclosure would work

## 3. Design System

### 3.1 Color Tokens (light, editorial — "paper and ink")

| Token | Hex | Role |
|---|---|---|
| `background` | `#F7F5F0` | Bone page background |
| `surface` | `#FFFFFF` | Cards, inputs |
| `foreground` | `#1C1917` | Primary text (warm ink) |
| `mutedForeground` | `#78716C` | Secondary text |
| `border` | `#E7E5E4` | Hairlines |
| `primary` | `#0D6B5C` | Deep emerald — CTAs, links |
| `primaryForeground` | `#F7F5F0` | Text on primary |
| `accent` | `#C45C4A` | Coral — highlights/warnings |
| `success` | `#0D6B5C` | Build ready / 90%+ |
| `warning` | `#B45309` | Incomplete profile |
| `destructive` | `#B91C1C` | Errors |
| `overlay` | `rgba(28,25,23,0.4)` | Modal/sheet backdrops |

Rules: no pure `#000`/`#fff`, no purple primaries, gradients only as soft bone→sand washes on marketing/walkthrough screens.

### 3.2 Typography

- **UI/Display font:** DM Sans (weights 400, 500, 700)
- **Mono/metrics font:** JetBrains Mono (400, 700)
- Scale ratio ≥ 1.25 between heading steps
- Body text max ~65 characters per line where possible
- No serif fonts on forms/settings/dashboard (serif allowed only as a *portfolio* font choice — Playfair Display)

### 3.3 Spacing & Radius

- Spacing scale: `4, 8, 12, 16, 24, 32, 48` (section gaps favor 24–32 on mobile)
- Radius: `sm=10`, `md=14`, `lg=20` (buttons/containers), `xl=28` (hero panels)
- Elevation: subtle tinted shadow only — `0 12px 32px -12px rgba(28,25,23,0.08)`

### 3.4 Layout & Motion

- Single column on phone. No nested cards. No 3-equal-column feature grids. No left-border accent stripes on list rows.
- Reanimated springs: `stiffness: 100, damping: 20` (general); onboarding step transitions use `damping: 22, stiffness: 210, mass: 0.85`
- Staggered `FadeInDown` on screen mount (60–80ms stagger between elements)
- Press feedback: `scale(0.98)` on primary buttons
- **Never fake progress** — build screen must reflect real `site_builds.status`

### 3.5 Haptics

| Event | Haptic |
|---|---|
| Taps, tab change | Light impact |
| OTP verified, build complete | Success |
| Profile < 90% on Home mount (once/session) | Warning |
| Form validation failure | Error |

### 3.6 Shared Components (build these first)

- `ScreenShell` — safe-area wrapper with consistent padding/background
- `ScreenHeader` — title + subtitle block used on auth/onboarding screens
- `OnboardingShell` — wraps onboarding screens with progress bar/step indicator + back button
- `BexoButton` — primary/secondary button variants, loading state, optional icon
- `FormField` — labeled text input with error state
- `OTPInput` — segmented OTP code entry with auto-advance
- `SkillTag` — toggleable pill for skill selection
- `LocationInput` — location autocomplete field
- `ProgressRing` — circular completion indicator
- `BuildStatusCard` — shows live build status (queued/building/done/failed)
- `ProfileActionStrip` — quick action row on dashboard
- `EmptyState` — empty list placeholder
- `ListRow` — generic list item row
- `YearPickerSheet` / `MonthPickerSheet` — bottom-sheet pickers for date entry

```sql
-- Enable UUID generation
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- ============================================================
-- PROFILES — one per authenticated user
-- ============================================================
CREATE TABLE IF NOT EXISTS public.profiles (
  id                      UUID        DEFAULT uuid_generate_v4() PRIMARY KEY,
  user_id                 UUID        REFERENCES auth.users(id) ON DELETE CASCADE NOT NULL UNIQUE,
  handle                  TEXT        UNIQUE, -- nullable until claimed
  full_name               TEXT        NOT NULL DEFAULT '',
  headline                TEXT        NOT NULL DEFAULT '',
  bio                     TEXT        NOT NULL DEFAULT '',
  avatar_url              TEXT,
  location                TEXT,
  website                 TEXT,
  linkedin_url            TEXT,
  github_url              TEXT,
  resume_url              TEXT,
  email                   TEXT,
  phone                   TEXT,
  is_published            BOOLEAN     NOT NULL DEFAULT FALSE,
  portfolio_theme         TEXT        NOT NULL DEFAULT 'default',
  portfolio_font          TEXT        NOT NULL DEFAULT 'modern',
  identity_card_palette   TEXT        NOT NULL DEFAULT 'midnight',
  identity_card_template  TEXT        NOT NULL DEFAULT 'standard',
  identity_card_font      TEXT,
  dob                     DATE,
  website_preference      TEXT,           -- stores selected "vibe" tags
  phone_verified          BOOLEAN     NOT NULL DEFAULT FALSE,
  email_verified          BOOLEAN     NOT NULL DEFAULT FALSE,
  consent_accepted_at     TIMESTAMPTZ,
  created_at              TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at              TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_profiles_user_id ON public.profiles(user_id);
CREATE INDEX IF NOT EXISTS idx_profiles_handle  ON public.profiles(handle);

-- Auto-create profile row on signup
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.profiles (user_id, full_name, avatar_url, email)
  VALUES (
    NEW.id,
    COALESCE(NEW.raw_user_meta_data->>'full_name', NEW.raw_user_meta_data->>'name', ''),
    NEW.raw_user_meta_data->>'avatar_url',
    NEW.email
  )
  ON CONFLICT (user_id) DO NOTHING;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

DROP TRIGGER IF EXISTS on_auth_user_created ON auth.users;
CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

  build_log      TEXT,

  updated_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()

-- ============================================================
-- ANALYTICS
-- ============================================================
CREATE TABLE IF NOT EXISTS public.site_analytics (
  id           UUID        DEFAULT uuid_generate_v4() PRIMARY KEY,

  user_agent   TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_analytics_profile ON public.site_analytics(profile_id);
CREATE INDEX IF NOT EXISTS idx_analytics_event   ON public.site_analytics(event_type);

-- ============================================================
-- updated_at TRIGGER HELPER
-- ============================================================
CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE OR REPLACE TRIGGER profiles_updated_at
  BEFORE UPDATE ON public.profiles
  FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

CREATE OR REPLACE TRIGGER site_builds_updated_at
  BEFORE UPDATE ON public.site_builds
  FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

-- ============================================================
-- ROW LEVEL SECURITY
-- ============================================================
ALTER TABLE public.profiles      ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.education     ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.experiences   ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.projects      ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.research      ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.skills        ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.updates       ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.site_builds   ENABLE ROW LEVEL SECURITY;
ALTER TABLE public.site_analytics ENABLE ROW LEVEL SECURITY;

-- Owner can manage their own rows; public can read PUBLISHED profiles + their children
CREATE POLICY "Users manage own profile" ON public.profiles
  FOR ALL USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);
CREATE POLICY "Public can read published profiles" ON public.profiles
  FOR SELECT USING (is_published = TRUE);

-- Repeat equivalent owner + public-read-if-parent-published policies for:
-- education, experiences, projects, research, skills, updates
-- (owner check via profile_id -> profiles.user_id = auth.uid())
```

## 6. State Management (Zustand Stores)

### 6.1 `useAuthStore`
Fields: `session`, `user`, `isLoading`, `phoneNumber`, `otpSentAt`, `hasSeenWalkthrough`, `dataConsentAccepted`, `collectedEmail`, `collectedPhone`.
Methods: `setPhoneNumber`, `setOtpSentAt`, `getOtpRemainingSeconds()` (10-min expiry), `setHasSeenWalkthrough`, `setDataConsentAccepted`, `setCollectedEmail`, `setCollectedPhone`.

### 6.2 `useProfileStore`
Fields: `profile` (maps to `profiles` table), `education[]`, `experiences[]`, `projects[]`, `skills[]`, `research[]`, `isLoading`, `onboardingStep`, `manualReviewStepIndex`, `parsedResumeData`.

Types:
```ts
type Education  = { id?: string; institution: string; degree: string; field: string; start_year: number; end_year?: number; gpa?: string };
type Experience = { id?: string; company: string; role: string; start_date: string; end_date?: string; description: string; is_current: boolean };
type Project    = { id?: string; title: string; description: string; tech_stack: string[]; live_url?: string; github_url?: string; image_url?: string };
type Research   = { id?: string; title: string; publication?: string; description: string; url?: string; year?: number };
```

Methods:
- `createProfile()`, `updateProfile(partial)`, `checkHandle(handle): Promise<boolean>`
- `setOnboardingStep(step)`, `setManualReviewStepIndex(i)`
- `setParsedResumeData(data)`
- `isOnboardingGateComplete(): boolean` — true once 90% completeness reached
- CRUD for education/experience/projects/skills/research arrays

**Onboarding step values** (used by root router to resume mid-flow): `email | photo | handle | resume | manual | manual_review | cards | about | dob | theme | font | preference | generating | completed`

### 6.3 `usePortfolioStore`
Fields: `buildStatus` (`queued|building|done|failed|null`), `portfolioUrl`, `buildLog`.
Methods: `subscribeToBuilds(profileId)` — opens a Supabase Realtime channel on `site_builds` filtered by `profile_id`, updates store on change. `triggerBuild()` — calls the build webhook.

## 7. Profile Completeness & Build Gate (the "90% rule")

The portfolio auto-builds once the profile crosses **90% completeness**. Both client and server MUST compute this the same way. Suggested weighting (sums to 100):

| Field group | Weight |
|---|---|
| `full_name`, `handle`, `avatar_url` | 15 |
| `headline`, `bio` (≥20 chars), `location` | 15 |
| `email`, `phone` (verified) | 10 |
| At least 1 education entry | 15 |
| At least 1 experience entry | 15 |
| At least 1 project entry | 15 |
| At least 3 skills | 10 |
| `portfolio_theme`, `portfolio_font` selected | 5 |

If experience is empty, the Dashboard shows a **persistent reminder** to upload a resume PDF (not just "add experience") — per product principle "resume PDF called out when work history is empty."

Build trigger logic:
- On `generating.tsx` mount, compute completion %. If ≥ 90% and no existing `site_builds` row in progress, call `tryAutoBuild()` → POST to build webhook → insert `site_builds` row with `status='queued'`.
- Subscribe to `site_builds` via Realtime; UI reflects `queued → building → done/failed` with real timestamps, no synthetic timers.
- On `done`, store `portfolio_url` on profile, set `is_published = true`, `onboardingStep = 'completed'`, haptic success, route to Dashboard.

## 8. Screen-by-Screen Specification

> Order = exact navigation order from first launch to main app. Each entry lists: route, purpose, fields/UI, validation, and next-step logic.

### 8.1 Root Router — `app/index.tsx`
**Type:** Invisible / logic-only screen (renders blank `background` colored view).
**Logic:**
```
if auth/profile loading → render blank
else if session exists:
    if onboardingStep === 'completed' OR isOnboardingGateComplete():
        set onboardingStep = 'completed' (if not already)
        → redirect to /(main)/(tabs)/dashboard
    else:
        step = onboardingStep || 'email'
        route = (step === 'manual_review') ? 'manual-review' : step
        → redirect to /(onboarding)/{route}
else:
    → redirect to /(auth)
```
This is what makes onboarding **resumable** — users can close the app mid-flow and pick up exactly where they left off.

### 8.3 Login — `(auth)/index.tsx`
**Purpose:** Phone-number entry + WhatsApp OTP request. Primary auth method.

**On mount:**
- If `!hasSeenWalkthrough` → redirect to `/(auth)/walkthrough`
- If session exists → redirect to `/dashboard`

**UI elements:**
- Centered Bexo logo (72×72)
- `ScreenHeader`: title = "Your work deserves\na beautiful home." / subtitle = "Create a stunning portfolio in seconds. Show the world what you're capable of."
- WhatsApp badge: green dot + "Fast login with WhatsApp" (bg `#25D36614`, border `#25D36633`, text `#1B7A4A`)
- Form:
  - Label: "Mobile number"
  - Country code selector (default `+91`) → opens modal picker with searchable list of country codes
  - Phone number text input (`keyboardType="phone-pad"`, `autoComplete="tel"`)
  - Inline validation error text in `destructive` color
  - Consent checkbox row: "I agree to BEXO processing my phone number and profile data as described in the **Privacy Notice** and **Terms**. Required before we send an OTP." — links open `mybexo.com/privacy` and `mybexo.com/terms`
  - Primary button: "Send me the code"

**Validation:**
- Phone must pass `isValidNationalNumber(phone, countryCode)` → else "Enter a valid phone number"
- `dataConsentAccepted` must be true → else "Please accept the data processing notice to continue."

**On submit:**
- `POST /auth/send-otp` with `{ phone: fullPhone }`
- On success: store `phoneNumber`, `otpSentAt = Date.now()`, navigate to `/(auth)/verify`
- On error: show sanitized error message inline

---

### 8.4 OTP Verify — `(auth)/verify.tsx`
**Purpose:** Verify the 4-digit WhatsApp OTP.

**UI:**
- `OTPInput` — 4 segmented boxes, auto-focus, auto-advance per digit, auto-submit when all 4 filled
- Countdown timer (10:00 → 0:00, format `m:ss`) showing OTP expiry
- "Resend code" link/button — disabled while timer > 0, re-enabled at 0, restarts timer on tap
- "Verify" button (also auto-triggers on 4th digit)
- Inline error display

**Logic:**
- `remaining = getOtpRemainingSeconds()`; ticks down every second via `setInterval`
- `expired = remaining <= 0` → disables verify button, shows "Code expired — resend"
- On successful verify: Supabase session created → root router takes over (decides onboarding vs dashboard)
- On resend: re-call send-otp endpoint, reset `otpSentAt`

### 8.5 Collect Phone — `(auth)/collect-phone.tsx`

## 9. Onboarding Flow (in order)

All onboarding screens are wrapped in `OnboardingShell`, which provides:
- Top progress bar (current step / total steps)
- Back button (navigates to previous step, persists `onboardingStep`)
- Consistent padding, `ScreenShell` background

Every screen calls `setOnboardingStep('<next-step>')` **before** navigating, so the root router can resume correctly if the app is killed mid-flow.

### 9.1 Email — `(onboarding)/email.tsx`

**UI:**

### 9.4 Resume Upload — `(onboarding)/resume.tsx`
**Step value:** `resume` → next: `cards` (AI path) OR `manual` (manual path)

**Purpose:** Upload a PDF resume; AI (GPT-4o via Supabase Edge Function) extracts structured data.

**UI states (`Stage` = `idle | uploading | parsing | done | error`):**

- **Idle:**
  - Illustration/icon + heading: "Upload your resume"
  - Subtext: explain AI will extract education, experience, skills, projects
  - "Choose PDF" button → `expo-document-picker`, restricted to `application/pdf`
  - Secondary link/button: **"Skip — enter manually instead"** → navigates to `(onboarding)/manual`, `setOnboardingStep('manual')`

- **Uploading:**
  - Selected filename shown
  - Progress bar (`uploadProgress`)

- **Parsing:**
  - Animated `ActivityIndicator`
  - Rotating status messages every 2.5s, cycling through:
    `["Scanning document...", "Extracting timeline...", "Formatting skills...", "Structuring education...", "Almost there..."]`

- **Done:**
  - Success checkmark + summary of extracted sections (e.g., "Found 2 education entries, 3 experiences, 5 projects, 12 skills")
  - "Continue" button → `cards`

- **Error:**
  - Friendly error message via `friendlyResumeAiError(err)`
  - "Try again" + "Enter manually instead" buttons (haptic `warning`)

**Logic:**
```ts
pickDocument() {
  request DocumentPicker (PDF only, copyToCacheDirectory: true)
  if canceled → return
  setStage('uploading')
  upload file to Storage bucket 'resumes' under user.id path
  setStage('parsing')
  call uploadAndParseResume(file) → Edge Function (GPT-4o) → ParsedResume
  setParsedResumeData(parsedResume)   // stored in profile store for cards.tsx prefill
  setStage('done')
  haptic success
}
```
**Continue (done state):** `setOnboardingStep('cards')` → push `/(onboarding)/cards`
**Back:** → `/(onboarding)/handle`

### 9.5 Manual Entry — `(onboarding)/manual.tsx`
**Step value:** `manual` → next: `manual_review`
**Only reached via the "Skip" link on Resume Upload.**

**Purpose:** Multi-step animated form replicating everything the AI would have extracted, entered by hand.

**Structure:** A horizontal-paging, step-indexed wizard (springs: `damping: 22, stiffness: 210, mass: 0.85` for screen transitions; `damping: 18, stiffness: 260, mass: 0.7` for in-step UI; `damping: 20, stiffness: 180` for the progress bar). Steps:

1. **About / Headline** — short professional headline (e.g., "Computer Science Student @ XYZ University")
2. **Education** (repeatable list, "Add another" pattern)
   - Institution, Degree, Field of study, Start year (via `YearPickerSheet`), End year (optional, "Currently studying" toggle), GPA (optional)
3. **Experience** (repeatable list)
   - Company, Role, Start month/year (via `MonthPickerSheet` + `YearPickerSheet`, stored as ISO `yyyy-mm-01`), End month/year OR "I currently work here" toggle, Description (multiline)
4. **Projects** (repeatable list)
   - Title, Description, Tech stack (tag input, comma-separated → `string[]`), Live URL (optional), GitHub URL (optional)
5. **Skills** — tag picker (reuse `SkillTag` + `SUGGESTED_SKILLS` list — see §9.7) plus free-text "add custom skill"
6. **Research / Publications** (optional, repeatable list, can be skipped entirely)
   - Title, Publication/venue, Description, URL, Year
   - `formatResearchPreview()` truncates description to 100 chars for list display

**Each repeatable section:**
- Empty state: "No entries yet" + "+ Add" button opens an inline form (not a separate screen)
- Each saved entry renders as a summary card with Edit/Delete (trash icon, haptic `tapLight` on delete with confirm)
- "Add another" button at bottom of list
- All fields **optional at the section level** — user can have zero experience/projects/research and still proceed (per "manual entry should not block on empty fields")

**Navigation within wizard:**
- Top progress bar fills proportionally to step index
- "Back" arrow (top-left, `Feather` icon) → previous step, or exits to `/(onboarding)/resume` if on step 1
- "Continue" / "Next" button at bottom — label changes to "Review" on final step
- On final "Review" tap: persist all arrays to `useProfileStore` (`education[]`, `experiences[]`, `projects[]`, `skills[]`, `research[]`), `setOnboardingStep('manual_review')`, push `/(onboarding)/manual-review`

**Date helpers required:**
```ts
pad2(n) → "0" + n if n < 10
monthIndexFromLabel(label) → index in MONTHS array + 1
isoYearMonthFirst(year, monthLabel) → `${year}-${pad2(monthIndex)}-01`
yearFromIso(iso) → first 4 chars if valid yyyy
monthLabelFromIso(iso) → MONTHS[monthIndex - 1]
```

### 9.13 Generating — `(onboarding)/generating.tsx`

  - Below 90%: "Almost there — finish your profile to unlock your site" (with a button back to the incomplete section)

```ts
const { completion, phase, statusLabel, tryAutoBuild, triggering } = useProfileBuildGate({ autoTrigger: false });

useEffect(() => {
```

## 12. Services Layer

### 12.1 `services/resumeParser.ts`
- `uploadAndParseResume(file, userId)`:
  1. Upload PDF to Supabase Storage `resumes/{userId}/{filename}`
  2. Invoke Supabase Edge Function (e.g., `parse-resume`) passing the storage path
  3. Edge Function downloads PDF, extracts text, sends to **OpenAI GPT-4o** with a structured-output prompt requesting JSON matching `ParsedResume`:
     ```ts
     type ParsedResume = {
       headline?: string;
       bio?: string;
       location?: string;
       education: Education[];
       experiences: Experience[];
       projects: Project[];
       skills: { name: string; category?: string; level?: string }[];
       research?: Research[];
     };
     ```
  4. Returns parsed JSON to client
- `friendlyResumeAiError(err)`: maps raw errors (timeout, malformed PDF, AI parse failure, rate limit) to user-friendly strings for the error UI state

### 12.2 `services/upload.ts`
- `uploadAvatar(userId, localUri)`: resizes/compresses image (via `expo-image-manipulator`), uploads to `avatars` bucket as `{userId}/avatar.jpg`, returns public URL

### 12.3 `services/achievementParser.ts`
- `parseUpdateText(text)`: lightweight NLP/heuristic (or AI call) to classify free-text post into `{ type, title, description }` for the Post Update tab
